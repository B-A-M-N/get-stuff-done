## Locked Decisions

Decision: The four CRITICAL static-only findings are false positives caused by over-specified policy requirements, not by missing runtime guards.
Decision: The actual guard implementations in `context.cjs` and `verify.cjs` satisfy the enforcement requirement; no additional guards in `gsd-tools.cjs` are needed.
Decision: The fix is to remove the `gsd-tools.cjs` entries from the four affected operations in `required-validators.yaml`.
Decision: After policy correction, re-running the audit must produce `final_status: VALID` with `summary.critical: 0`.
Decision: The `76-VERIFICATION.md` will be regenerated via `audit enforcement-boundary --write`.
Decision: The `76-VALIDATION.md` must be manually updated to `nyquist_compliant: true` to close the phase.

## Risks

Risk: Removing gsd-tools.cjs requirements might hide a real enforcement gap if those routes can bypass the command-module guards.
Guardrail: Verify that all four workflows are indeed routed through the command modules and cannot reach the guard-less paths.
  - `context build --workflow plan-phase` → `cmdContextBuild` (context.cjs)
  - `context build --workflow execute-plan` → `cmdContextBuild` (context.cjs)
  - `verify integrity` → `cmdVerifyIntegrity` (verify.cjs)
  - `verify workflow-readiness` → `cmdVerifyWorkflowReadiness` (verify.cjs)
Fallback: if any workflow can be invoked without hitting the guard, we must add the appropriate routing guard rather than remove the policy requirement.

## Assumptions

Assumption: The command routing in `gsd-tools.cjs` sends all four workflows to their respective command module functions, and those functions contain the degraded-mode checks. This is currently true.
Assumption: No other code paths (e.g., direct library calls) can invoke these workflows without passing through the same checks. The guards are at the command entry points, which is sufficient.
Assumption: The runtime probes already confirm the expected blocking behavior, so the guards are functionally correct.
