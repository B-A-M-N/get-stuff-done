Phase 76.1 must reconcile:

- **Static classifier output**: `audit enforcement-boundary --raw` returns `INVALID` with 4 CRITICAL static-only findings
- **Sanctioned runtime probe evidence**: The same audit's embedded runtime probes disprove those bypasses (probes return `disproven` with expected `success: false`)

<decisions>
## Implementation Decisions

### Root cause
The static classifier in `get-stuff-done/bin/lib/enforcement-boundary-audit.cjs` searches for required guard patterns (e.g., `enforceWorkflowOrBlock(cwd, 'context:plan-phase')`) only in `gsd-tools.cjs`. However, the actual guard implementation lives in the command modules (`context.cjs`, `verify.cjs`) and does not use that helper function at all.

Result:
- Static scan misses real guards → false-critical findings
- Runtime probes run the actual commands and correctly observe blocking behavior
- Discrepancy makes the audit surface untrustworthy

### Scope boundaries
In scope for Phase 76.1:
- Update the static classifier to recognize distributed guard implementations
- OR adjust the final-status synthesis to not treat static-only CRITICALs as definitive when contradicted by sanctioned runtime probes
- Ensure the audit artifact's `final_status` reflects the **authoritative truth** (no critical bypass exists) rather than a heuristic mismatch

Out of scope:
- Changing the actual guard implementations (they are correct)
- Broadening the audit to unrelated surfaces
- Modifying runtime probe definitions (they are working as designed)

### Invariant-driven closure
Phase 76.1 must establish explicit invariants for enforcement-boundary truth:

- `static_classifier_coverage`: Classifier must recognize all guard patterns used in the codebase, OR final status must account for pattern gaps
- `runtime_probe_authority`: Runtime probe results are authoritative for Bypass existence determination
- `final_status_consistency`: Final `VALID/INVALID` must match the ground truth (no known critical bypass)

These invariants must be codified in the audit logic and validated by this phase.

### Acceptable resolutions
The gap can be closed by **either**:

1. **Classifier enhancement** — Extend `classifyValidatorBypasses` to also check command modules (`context.cjs`, `verify.cjs`, etc.) for direct `degradedMode.evaluateWorkflow()` calls with the required workflow keys.
2. **Severity reconciliation** — Introduce a new finding status like `STATIC_ONLY` and adjust final status calculation: if any `STATIC_ONLY` finding has a corresponding runtime probe that `disproven` the bypass, the final status should not count it toward `critical` for `INVALID` determination.
3. **Policy alignment** — Explicitly state in policy that Runtime proof supersedes static classification, and update the artifact synthesis accordingly.

Any of these approaches must be reflected in code changes and validated by regenerated audit output.

### Proof surface
The authoritative audit surface is:

```bash
node get-stuff-done/bin/gsd-tools.cjs audit enforcement-boundary --write --raw
```

Phase 76.1 is complete when:

- `final_status` is `VALID`
- `summary.critical` is `0`
- All four current static-only findings are either:
  - Resolved by classifier now detecting the actual guard patterns, **or**
  - Downgraded to non-critical status with explicit runtime probe evidence showing bypass disproven
- The generated `76-VALIDATION.md` shows `nyquist_compliant: true`

### Dependencies
- Phase 76 (original enforcement boundary audit) must be complete; this phase repairs its closure
- No dependence on later phases (77, 78, 79, 80, 80.1, 81)
