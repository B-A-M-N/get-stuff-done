# Phase 80.1 Research: Memory Truth Reproval Closure

## Summary

Phase 80.1 should be rerun as a three-pass closure phase:
- Pass A: audit Phase 75 truth synthesis against an explicit invariant contract
- Pass B: repair only the missing schema, logic, and truth inputs that keep the memory-truth enforcement area conditional
- Pass C: explicitly regenerate verification, then truth, then compare them and fail if they diverge

Primary recommendation:
- add an authoritative invariant file for Phase 75 rather than continuing to infer closure predicates from generated truth artifacts
- emit one machine-readable audit artifact that identifies exactly which invariants are failing or missing
- treat correct fail-closed degraded memory behavior as `VALID`, not `CONDITIONAL`
- make planning-path blocked-memory proof part of the same enforcement area instead of treating it as ambient system posture
- use explicit regeneration commands for closure and reject side-effect-only truth refreshes
- keep Phase 74 work sequential and follow-on: Phase 75 must go `VALID` first, then Phase 74 can be re-closed for `TRUTH-DRIFT-02`

The repo already has the behavioral surfaces needed for this work. What is missing is an explicit invariant contract and a deterministic closure path that both verification and truth synthesis consume.

## Architecture Patterns

### 1. Invariant file as authoritative predicate source

Recommended artifact:
- `.planning/phases/75-degraded-mode-enforcement/75-INVARIANTS.yaml`

Recommended use:
- define each invariant explicitly
- define required evidence surfaces per invariant
- define whether each invariant feeds final truth synthesis
- make both verification and truth-generation logic consume or at least align to this contract

This avoids predicate drift where verification and phase truth reach different conclusions from different implicit rules.

### 2. Audit before repair

Recommended artifact:
- `.planning/phases/80.1-memory-truth-reproval-closure/80.1-A-AUDIT.json`

Recommended behavior:
- evaluate each invariant against current direct evidence
- classify each as `PASS`, `FAIL`, or `MISSING`
- identify exact blocker type:
  - schema
  - logic
  - missing input

This keeps repair bounded. The repo should know why truth is conditional before changing logic.

### 3. Treat fail-closed degraded memory as successful closure

Recommended synthesis rule:
- if canonical Postgres memory is unavailable
- and model-facing memory is blocked
- and degraded state is explicitly surfaced
- then the memory enforcement area is `VALID`

The current `CONDITIONAL` result is only correct when truth is ambiguous, not when the system is behaving correctly under degraded conditions.

### 4. Planning-path proof is part of the same enforcement area

Recommended authoritative planning proof surfaces:
- `context build --workflow plan-phase --raw`
- `context build --workflow execute-plan --raw`

These must prove that planning flows do not silently regain model-facing memory truth while canonical memory is unavailable.

### 5. Explicit regeneration beats side effects

Recommended closure order:
1. regenerate `75-VERIFICATION.md`
2. regenerate `75-TRUTH.yaml`
3. compare statuses and invariant results
4. fail if either remains non-`VALID`

This should be implemented with explicit commands or helpers. Do not rely on verifier side effects to claim closure.

### 6. Sequential Phase 74 coupling

Phase 74 remains in scope for Phase 80.1 completion because `TRUTH-DRIFT-02` is mapped here, but it should stay downstream of Phase 75 truth normalization.

Recommended posture:
- do not attempt simultaneous closure
- first eliminate the Phase 75 truth split
- then use the same explicit regeneration discipline to close the residual Phase 74 drift reproval gap

- only then proceed to Phase 74 residual drift closure

### Strongly recommended

- keep invariant evaluation machine-readable and separate from narrative verification
- add focused tests for any new synthesis or invariant logic instead of widening broad test suites
- encode the explicit regeneration order in plan acceptance criteria
- emit `80.1-BLOCKERS.md` whenever closure still fails after repair

### Out of scope

- milestone-wide truth cleanup
- Nyquist backfill and bookkeeping closure assigned to Phase 81
- generalized drift-engine redesign
- redefining `CONDITIONAL` to include correct degraded fail-closed behavior

## Common Pitfalls

- accepting requirement-level closure while phase truth remains conditional
- inferring invariants from generated truth instead of authoring them explicitly
- letting planning-path memory blocking remain ambient or implied
- relying on verifier side effects to claim final closure
- broadening 80.1 into unrelated system health cleanup

## Open Questions / Assumptions

- Assumption: Phase 75 can reach full `VALID` without a broad redesign once invariant-driven synthesis is introduced.
- Assumption: the remaining drift inputs needed by Phase 75 are narrow freshness/presence issues, not a separate large architectural bug.
- Open question for implementation: whether invariant evaluation should live directly in `phase-truth.cjs`, in a narrow helper adjacent to it, or in a shared helper consumed by both verification and truth generation.

## Don't Hand-Roll

- do not preserve the old two-pass “memory then drift” plan framing
- do not treat `75-TRUTH.yaml` as self-justifying
- do not let a `CONDITIONAL` phase truth artifact coexist with a `VALID` requirement verdict for the same enforcement area
- do not use narrative reasoning where a machine-readable blocker matrix can be emitted
