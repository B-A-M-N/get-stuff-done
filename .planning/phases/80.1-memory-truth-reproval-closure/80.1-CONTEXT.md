# Phase 80.1: Memory Truth Reproval Closure - Context

**Gathered:** 2026-03-28
**Status:** Ready for planning rerun

<domain>
## Phase Boundary

Phase 80.1 is a truth-closure normalization phase for the model-facing memory enforcement area. Its job is to eliminate the specific inputs that keep Phase 75 truth conditional even after runtime fail-closed memory behavior has been re-proven.

Phase 80.1 must close the split between:
- requirement-level evidence showing model-facing memory fails closed
- phase-level truth synthesis still classifying the same enforcement area as `CONDITIONAL`

This phase does not perform broad milestone cleanup. It repairs only the truth inputs, synthesis criteria, and explicit regeneration path required for authoritative closure of the same enforcement area.

</domain>

<decisions>
## Implementation Decisions

### Core truth rule
- No requirement-level truth may be marked closed if any phase-level truth artifact for the same enforcement area remains non-`VALID`.
- For this phase, authoritative closure requires:
  - `.planning/phases/75-degraded-mode-enforcement/75-VERIFICATION.md` = `VALID`
  - `.planning/phases/75-degraded-mode-enforcement/75-TRUTH.yaml` = `VALID`
- Any disagreement between those two artifacts is a hard failure for Phase 80.1.

### Enforcement-area definition
- The enforcement area for Phase 80.1 is `Model-Facing Memory Truth Closure`.
- Included:
  - runtime model-facing memory blocking
  - planning-path blocking caused by unavailable canonical trusted memory
  - degraded-state signaling for model-facing memory
  - any drift inputs actually consumed by Phase 75 truth synthesis for this memory closure
- Excluded:
  - unrelated global system health
  - unrelated milestone bookkeeping
  - unrelated Plane/Open Brain/integration cleanup unless they directly feed Phase 75 truth synthesis

### Invariant-driven closure
- Phase truth must be decomposed into explicit invariants instead of relying on a monolithic final verdict.
- Phase 80.1 must introduce:
  - `.planning/phases/75-degraded-mode-enforcement/75-INVARIANTS.yaml`
- This invariant file is the authoritative predicate inventory for memory-truth closure and must not be inferred from generated truth or verifier prose.
- Minimum invariant set:
  - `memory_blocking`
  - `planning_memory_blocking`
  - `degraded_state_signaling`
  - `drift_input_validity`
  - `verification_integrity`

### Audit artifact contract
- The truth-synthesis audit must emit one machine-readable blocker matrix:
  - `.planning/phases/80.1-memory-truth-reproval-closure/80.1-A-AUDIT.json`
- Required shape:
  - invariant name
  - `PASS|FAIL|MISSING`
  - evidence list
  - blocking reason
  - whether the invariant affects final truth synthesis
- Prose-only audit output is not sufficient.

### Planning-path proof surface
- Phase 80.1 must prove blocked memory truth on all authoritative planning workflows used by Phase 75 truth synthesis:
  - `context build --workflow plan-phase --raw`
  - `context build --workflow execute-plan --raw`
- Additional readiness or integrity surfaces are only in scope if they contribute directly to Phase 75 truth synthesis for this enforcement area.
- If any authoritative planning workflow still presents model-facing memory as usable under canonical-memory loss, the invariant fails.

### Drift boundary
- Phase 80.1 may only repair drift inputs to the extent required for Phase 75 truth synthesis to classify this enforcement area correctly.
- Allowed drift work:
  - ensure required drift inputs are present
  - ensure they are fresh
  - ensure they are machine-readable
- Not allowed:
  - broad drift-engine redesign
  - unrelated reconciliation behavior changes

### Degraded semantics
- Fail-closed degraded behavior is `VALID`, not `CONDITIONAL`, when all are true:
  - canonical Postgres-backed trusted memory is unavailable
  - model-facing memory is blocked
  - degraded state is surfaced explicitly
- `CONDITIONAL` is reserved for ambiguous or incompletely surfaced truth, not for correct fail-closed degraded behavior.

### Repair order
- Phase 80.1 follows this order:
  1. schema and invariant contract
  2. synthesis logic
  3. missing truth inputs
- Do not patch final classification first.

### Closure regeneration contract
- Closure must use explicit regeneration order:
  1. regenerate verification
  2. regenerate truth
  3. compare outputs
- Side-effect regeneration does not count as authoritative closure.
- If closure still fails, Phase 80.1 must emit:
  - `.planning/phases/80.1-memory-truth-reproval-closure/80.1-BLOCKERS.md`
- That blocker report must classify each remaining failure as:
  - schema
  - logic
  - missing input

### Cross-phase sequencing
- Phase 75 must reach `VALID` first.
- Phase 74 remains required for overall Phase 80.1 completion because `TRUTH-DRIFT-02` is still mapped here.
- Phase 74 closure is sequential, not simultaneous:
  - first Phase 75 memory-truth closure
  - then Phase 74 drift closure

### Claude's Discretion
- Exact schema fields inside `75-INVARIANTS.yaml` and `80.1-A-AUDIT.json`, as long as each invariant is independently auditable.
- Exact code location for invariant evaluation and truth comparison helpers.
- Exact explicit command sequence for truth regeneration, as long as verification is regenerated before truth and side effects are not relied on for closure.

</decisions>

### Unresolved Ambiguities

- None. Closure target, invariant source, audit format, planning proof surface, drift boundary, degraded semantics, regeneration order, failure reporting, and enforcement-area scope are locked for planning.

### Interpreted Assumptions

- The current runtime behavior is already close to correct for degraded memory fail-closed posture.
- The remaining issue is primarily truth-contract and truth-input normalization, not proof absence alone.
- Phase 75 truth currently stays conditional because required memory-truth inputs are either too coarse, missing, or regenerated through non-authoritative side effects.

<specifics>
## Specific Ideas

- Requirement truth and phase truth must converge, not merely appear compatible by human interpretation.
- Phase 80.1 is complete only when the system can say `VALID` mechanically for the same enforcement area.
- The audit output should identify which invariant is blocking closure before any repair begins.
- Correct degraded fail-closed behavior should harden truth, not downgrade it.

</specifics>
