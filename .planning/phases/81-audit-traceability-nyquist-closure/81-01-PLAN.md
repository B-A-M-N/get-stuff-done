## Locked Decisions

Decision: This plan READ-ONLY. Zero file mutations except summary and audit JSON outputs.
Decision: Truth Admissibility Contract (deterministic, no judgment):
  - VALID_ACCEPTED: final_status == VALID, passes truth validation, no conflicts
  - CONDITIONAL_ACCEPTED: CONDITIONAL AND ALL:
      * No requirement blocked (all mapped requirements closed by later VALID phase or self)
      * No downstream conflict (doesn't contradict Phase 79 gauntlet or Phase 80/80.1 truth)
      * Condition is stable (not introduced by Phase 80/80.1 changes, not newly degraded)
      * Condition is explicit (reason documented in gaps/status_reason, not implicit missing)
  - REPAIR_REQUIRED: INVALID OR (CONDITIONAL AND any blocker) OR inconsistent/malformed
  - MISSING: artifact does not exist
Decision: Inventory covers the full artifact set for phases 70, 71, 72, 73, 74, 75, 76, 77, 78, 79, 80, 80.1, 81 (39 artifacts).
Decision: Phase 81-02 MUST consult this inventory and ONLY operate on items where admissibility == REPAIR_REQUIRED or admissibility == MISSING.
Decision: Inventory must be machine-readable JSON to enable automated gating.

## Risks

Risk: validator commands might have different exit codes for VALID vs CONDITIONAL.
Guardrail: use `gsd-tools verify verification-artifact --raw` for verification (returns "valid" string); for truth artifacts **read TRUTH.yaml directly** (committed truth is authoritative, not regenerated); for validation check grep 'nyquist_compliant: true'.
Fallback: if YAML parsing fails or required fields missing, treat as STALE for safety.

Risk: some phases might have VERIFICATION.md but missing TRUTH/VALIDATION.
Guardrail: inventory each artifact type separately; a phase can be PARTIAL if some artifacts present and others missing.

Risk: stale STATE.md/ROADMAP.md could mislead inventory expectations.
Guardrail: inventory from actual files on disk, not roadmap claims.

## Assumptions

Assumption: validator commands are functional in current environment.
Assumption: all target artifact paths are known and stable.

    - phase, type, path, required: true
    - exists (boolean)
    - technical_status: for VERIFICATION/VALIDATION: "VALID"/"INVALID"/"ERROR"; for TRUTH: "VALID"/"CONDITIONAL"/"INVALID"
    - technical_classification: one of "VALID_EXISTING", "MISSING", "STALE", "INVALID", "VALID_UNVERIFIED"
    - admissibility: one of "VALID_ACCEPTED", "CONDITIONAL_ACCEPTED", "REPAIR_REQUIRED", "MISSING"
    - justification: array of strings (why this admissibility) — empty for VALID_ACCEPTED/MISSING
    - evidence: raw validator output or error
  - All 39 entries have final admissibility.
  </acceptance_criteria>
  <behavior>
    - For each target artifact:
      1. Check file exists → record exists.
      2. If exists, probe based on type:
         * VERIFICATION → `gsd-tools verify verification-artifact <path>` → parse JSON: .valid (bool), .phase_truth.final_status (string)
         * TRUTH → read the TRUTH.yaml file directly from its path (from inventory entry). Parse YAML, extract .final_status. Do NOT call phase-truth generate (committed truth is authoritative).
         * VALIDATION → check file contains 'nyquist_compliant: true' → set nyquist_compliant boolean
      3. Compute technical_classification:
         - VALID_EXISTING: exists AND appropriate validator passes (VERIFICATION: .valid==true; TRUTH: .final_status=="VALID"; VALIDATION: nyquist_compliant==true)
         - MISSING: file does not exist
         - INVALID: exists AND validator explicitly fails (VERIFICATION: .valid==false; TRUTH: .final_status=="INVALID" or .final_status=="CONDITIONAL"; VALIDATION: nyquist_compliant==false)
         - STALE: exists but validator error/parse failure
         - VALID_UNVERIFIED: exists but couldn't validate (no command run, non-JSON output)
      4. Compute admissibility based on artifact type:
         * For VERIFICATION & VALIDATION:
           - If technical_classification == VALID_EXISTING → admissibility = VALID_ACCEPTED
           - Else → admissibility = REPAIR_REQUIRED (any non-valid requires repair)
         * For TRUTH:
           - If technical_classification == VALID_EXISTING → admissibility = VALID_ACCEPTED
           - Else if technical_classification == MISSING → admissibility = MISSING
           - Else if final_status == "CONDITIONAL":
                Evaluate deterministically:
                a) Requirement Block Check:
                   - Read .planning/REQUIREMENTS.md.
                   - Locate the traceability table (section "## Traceability"). The table has columns: Requirement, Final Phase, Status.
                   - For each data row after the header (lines starting with '|'):
                       Split by '|' and trim to get cells.
                       The cell under "Final Phase" contains a string like "Phase 70". Extract the phase number by removing "Phase " prefix.
                       If the extracted phase number equals the current phase (as string), then this requirement is assigned to this phase.
                       Check the "Status" cell: if it is not "Complete", then this requirement is not closed → blocked = true.
                   - If no rows map to this phase, blocked = false (no requirements assigned).
                   - If all mapped rows have Status == "Complete", blocked = false.
                b) Downstream Conflict Check:
                   - Automatic contradiction detection infeasible → downstream_conflict = false.
                   - Core integrity enforced separately (Task 3).
                c) Condition Explicit Check:
                   - explicit = ( (.gaps // empty) | length > 0 ) or ( (.status_reason // empty) | length > 0 )
                d) Condition Stable Check:
                   - stable = true
                   - If .gaps is an array, iterate: if any .gaps[].type == "regression" or .gaps[].type == "new_degradation", then stable = false.
                IF (blocked == false AND downstream_conflict == false AND explicit == true AND stable == true):
                    admissibility = CONDITIONAL_ACCEPTED
                    justification = ["No requirement block", "Condition explicit"]
                    if stable → add "Condition stable"
                ELSE:
                    admissibility = REPAIR_REQUIRED
                    justification = []
                    if blocked → add "Requirement block"
                    if !explicit → add "Condition not explicit"
                    if !stable → add "Condition unstable"
                    if downstream_conflict → add "Downstream conflict"
           - Else (any other: INVALID, VALID_UNVERIFIED, STALE) → admissibility = REPAIR_REQUIRED
      5. Record all fields: technical_status, technical_classification, admissibility, justification, evidence.
  </behavior>
