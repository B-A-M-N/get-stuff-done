## Locked Decisions

Decision: `drift_catalog.yaml` is the authoritative baseline scope, but Phase 73 may report `missing_surface` and `untracked_surface` findings when runtime truth diverges from catalog scope.
Decision: Phase 73 may detect, severity-classify, and predict impact, but it must not mutate truth-bearing state artifacts.
Decision: Strong contradiction is required for strong drift claims; partial or unavailable evidence must degrade into `insufficient_evidence` or lower-confidence findings instead of overstating drift.
Decision: `CRITICAL` drift is blocking for exit behavior; `MAJOR` and `MINOR` remain zero-exit findings.
Decision: The latest report must carry stable metadata including generation time and baseline identity so later readers can attribute and trust the scan result.

## Risks

Risk: the engine can over-classify runtime unavailability as drift and poison trust in the report.
Guardrail: classify unavailability as `degraded_state` or `insufficient_evidence` unless contradiction with expected truth is directly observed.
Fallback: when evidence is partial, emit conservative findings with explicit confidence notes rather than upgrading to active drift.

## Assumptions

Assumption: the first stable report schema includes `generated_at`, `catalog_hash`, summary data, findings, and predicted-impact fields, and later schema refinement should extend this contract rather than replace it.
Open Question: exact helper boundaries between scan orchestration, report serialization, and status rendering can stay implementation-flexible so long as one canonical report object remains the source of truth.
