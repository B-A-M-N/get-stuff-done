## Locked Decisions

Decision: `drift status` is the canonical operator surface, while `drift scan` executes the engine and `drift report` exposes machine-readable report data.
Decision: `.planning/drift/latest-report.json` is the primary runtime artifact and is not a checked-in planning document.
Decision: Operator health must reflect active drift truth, but Phase 73 still must not mutate roadmap, verification, or state artifacts.
Decision: Only `CRITICAL` drift findings should drive a non-zero scan exit code; `MAJOR` and `MINOR` remain zero-exit surfaces.
Decision: If the latest report is missing or stale, `drift status` and health must surface that explicitly instead of implying healthy state.

## Risks

Risk: CLI surfaces can drift from the underlying engine and create a new operator-truth split.
Guardrail: `status` and `report` must render directly from the latest machine report rather than re-deriving their own view.
Fallback: if no runtime report exists, `status` should either trigger a scan or explicitly report stale/missing report state instead of inventing health.

## Assumptions

Assumption: the cleanest first integration point for operator truth is to have `drift status` and health surfaces read one shared latest-report artifact rather than each performing their own scan.
Open Question: whether drift visibility lands first in `brain health`, `health degraded-mode`, or both can remain implementation-flexible so long as at least one existing operator-health surface reports active drift truth in this phase.
