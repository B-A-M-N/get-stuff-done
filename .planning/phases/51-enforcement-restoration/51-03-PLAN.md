- No performance degradation (expected O(n) over small set)
</acceptance_criteria>

<action>
In `get-stuff-done/bin/lib/state.cjs`, within `cmdStateAddDecision`:

1. Replace existing string-equality check with hash-based comparison:
   - Implement `function computeHash(str)` using djb2 (or similar):
     ```javascript
     function computeHash(str) {
       let hash = 5381;
       for (let i = 0; i < str.length; i++) {
         hash = ((hash << 5) + hash) + str.charCodeAt(i); // hash * 33 + c
       }
       return hash >>> 0; // ensure unsigned
     }
     ```
   - Normalize the new entry: `const normalized = entry.trim().toLowerCase(); const newHash = computeHash(normalized);`
   - For each existing entry line, normalize and compute its hash; store in a Set for O(1) lookup. If `newHash` exists in set, treat as duplicate.
2. Keep the same output and early return behavior.

</action>

## Notes

- Deduplication hash: djb2 is sufficient; no need for cryptographic strength. Determinism across Node versions is fine (32-bit int).
- `.continue-here` should be overwritten on each pause; same filename is fine.
- The `workflow` namespace in `loadConfig` is additive; existing code using top-level keys (e.g., `config.auto_advance`) remains functional.
- Parsing of Performance Metrics should be tolerant of variations in markdown table formatting (pipes, alignment).
- The existing `cmdStateAddDecision` already has deduplication via string comparison; this task upgrades it to hash-based to satisfy requirement specification.
- The existing `cmdStatePause` and `cmdStateResume` need enhancement to use `.continue-here` and frontmatter status; current implementation only modifies body.
