# v0.7.0 Decision Set: Truth Enforcement & Drift Elimination

**Prepared:** 2026-03-27
**Status:** Authoritative draft for next-milestone definition
**Intent:** Convert the raw milestone megaprompt and ambiguity discussion into a clean decision set aligned to the actual GSD operating model.

## Milestone Boundary

`v0.7.0` is a hardening milestone. It does not add new user-facing capability. It tightens proof, verification, drift detection, degraded-mode behavior, and enforcement boundaries so the system cannot report state that it cannot prove through observable artifacts and enforced execution paths.

## Milestone Thesis

The system must never report a state that cannot be proven through:

- filesystem artifacts
- git history
- executable output
- or deterministic test evidence

Anything else is a claim, not truth.

## Locked Milestone Decisions

### Evidence Standard

- Allowed evidence is limited to:
  - filesystem evidence: file existence, file content, file hash
  - git evidence: commit hash, diff, changed-file list
  - execution evidence: CLI output, exit codes, structured logs
  - test evidence: deterministic passing output or machine-readable report
- Natural-language narrative is never sufficient evidence by itself.
- If no evidence artifact exists for a claim, the claim is `INVALID`.

### Drift Definition

- `Spec Drift`: roadmap or requirements claim a capability that code does not implement.
- `Implementation Drift`: code exists but behavior does not satisfy the requirement.
- `Verification Drift`: verification claims success without supporting evidence.
- `Execution Drift`: execution summary claims work completed without matching commits or artifacts.
- Any confirmed drift is a system inconsistency and must be surfaced explicitly.

### Drift Handling

- Drift must be classified as `CRITICAL`, `MAJOR`, or `MINOR`.
- `CRITICAL` drift blocks milestone completion.
- `MAJOR` drift requires explicit repair or downgrade before closure.
- `MINOR` drift may be documented but cannot be hidden.
- When drift is found, affected status must be downgraded mechanically:
  - `VALID` -> `CONDITIONAL`
  - `CONDITIONAL` -> `INVALID`
  - `VALID` -> `INVALID` directly when evidence is absent or contradicted

### Verification Standard

- `VERIFICATION.md` is an evidence artifact, not a narrative wrap-up.
- Every verification artifact must include:
  - `Observable Truths`
  - `Requirement Coverage`
  - `Anti-Pattern Scan`
  - `Drift Analysis`
  - `Final Status`
- Requirement coverage rows must use only:
  - `VALID`
  - `CONDITIONAL`
  - `INVALID`
- Anti-pattern scanning must check for at least:
  - `TODO`
  - `FIXME`
  - stubs
  - mocked logic presented as real behavior
  - unimplemented or dead branches

### Execution Proof Chain

- Every completed task must produce at least one git commit.
- A task is not complete unless its commit can be mapped to the task scope.
- `SUMMARY.md` must map execution claims to commits and touched files.
- “Already implemented” is not enough; existing code still requires evidence mapping.
- “No changes required” is not valid completion unless the task is explicitly reframed as verification-only and still produces a verification artifact with evidence.

### Degraded Mode Policy

- No silent fallback is allowed.
- Degraded systems must be visible and behaviorally meaningful.
- Degraded levels are:
  - `HEALTHY`
  - `DEGRADED`
  - `UNSAFE`
- `DEGRADED` must warn and reduce capability.
- `UNSAFE` must block affected operations.
- A subsystem may not present itself as healthy if required dependencies are unavailable.

### Memory Truth Policy

- Second Brain remains execution-truth authority for workflow memory.
- Open Brain remains optional long-horizon semantic recall.
- If canonical Postgres-backed memory is unavailable, model-facing memory must fail closed rather than masquerade as trusted planning input.
- SQLite fallback may remain useful for bounded operational behavior where already sanctioned, but it must not be silently treated as equivalent truth for model-facing planning context.

### Authority Model

- Planner authority: intent definition and decomposition.
- Executor authority: implementation and artifact production.
- Verifier authority: truth judgment.
- Verifier may downgrade planner or executor claims when evidence is absent, partial, or contradictory.

### Phase Completion Standard

- A phase is complete only when:
  - all plans for the phase are executed or explicitly retired
  - each completed task has commit-backed proof
  - `VERIFICATION.md` exists and has final status `VALID`
  - no `CRITICAL` drift remains in the phase scope
- Working behavior without proof is treated as `INVALID`, not “probably done.”

### Failure Handling

- Integrity failures are blocking failures, not soft warnings.
- On blocking failure the system must:
  - stop the affected flow
  - emit a failure classification
  - produce or request a repair plan
- “Best effort” completion is not valid for this milestone’s control surfaces.

### Bypass Definition

- A bypass is any path that:
  - skips required CLI primitives or validators
  - writes critical state without the sanctioned interface
  - reports success without proof artifacts
  - modifies truth-bearing files without traceable execution evidence
- Bypass paths are treated as enforcement failures and must be audited directly.

### Operator Visibility

- Operators must be able to inspect:
  - system health
  - drift status
  - verification status
  - execution history
- Operator surfaces must reflect actual backend truth, not optimistic or inferred state.

### Parallelism Policy

- Milestone phases are sequential by dependency unless independence is explicitly proven.
- Plans within a phase may run in parallel by wave when all of the following are true:
  - file isolation: they do not mutate the same files
  - artifact isolation: they do not write the same summary, verification, or state artifact
  - truth isolation: they do not both define or mutate the same source of truth
  - deterministic mergeability: proof chains remain attributable without ambiguity
  - independent verification: each plan can close with its own evidence before phase aggregation
- Integrity constrains parallelism; it does not ban it.

### Testing Contract

- Tests used as truth evidence must be deterministic.
- Test output should be machine-readable when practical.
- Test coverage must map back to requirement or enforcement surfaces, not just generic code paths.

### Allowed Assumptions

- Assumptions are only allowed when:
  - they are explicitly documented
  - they are labeled as conditional
  - they do not get promoted to truth without later evidence
- Undocumented assumptions are invalid inputs to verification.

## Non-Goals

`v0.7.0` does not exist to:

- improve Open Brain recall quality
- optimize embeddings
- add new user-facing capability
- do generic UX polish

Those belong in later milestones once truth surfaces are mechanically trustworthy.

## Expected Phase Direction

The current milestone concept is expected to decompose roughly as:

- Phase 70: drift surface mapping
- Phase 71: execution proof chain
- Phase 72: verification hardening
- Phase 73: drift detection engine
- Phase 74: state reconciliation layer
- Phase 75: degraded-mode enforcement
- Phase 76: enforcement boundary audit
- Phase 77: phase truth contracts
- Phase 78: end-to-end integrity gauntlet

This list is directional, not yet authoritative roadmap text. The next milestone-definition flow should convert it into requirements and phased scope.

## Open Items For Milestone Definition

- Whether truth contracts should be represented as a single shared schema, multiple artifact-specific schemas, or both.
- Whether drift detection ships as one CLI surface or a small cluster of narrower commands behind a common parent.
- Which existing verification and audit artifacts should be retrofitted first versus grandfathered and revalidated over time.

These are implementation questions for the milestone-definition and phase-discussion steps, not unresolved thesis ambiguity.
